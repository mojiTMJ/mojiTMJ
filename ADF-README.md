## 🚀 Project Overview

This repository stores all Azure Data Factory artefacts (pipelines, datasets, linked services, triggers, data flows) as JSON code, enabling:

- **Full version control** of every ADF change
- **Code review** via pull requests before any deployment
- **Automated CI/CD** to Dev → QA → Production environments
- **Environment parameterisation** – one codebase, three environments

## 🔗 Connect ADF to This Repository

1. Open your Azure Data Factory instance in the Azure Portal
2. Go to **Manage** → **Git configuration**
3. Select **GitHub** as the repository type and authenticate
4. Choose this repository
5. Set the following:
   | Setting | Value |
   |---------|-------|
   | Collaboration branch | `dev` |
   | Publish branch | `adf_publish` |
   | Root folder | `/` |
6. Click **Apply**

From this point, all ADF changes saved in the UI are committed to branches in this repository.

## 🌿 Branch Strategy

| Branch | Purpose | Protection |
|--------|---------|------------|
| `main` | Production-ready code | ✅ Required reviews, required status checks, no force push |
| `qa` | QA / staging | ✅ Required reviews |
| `dev` | Active development (ADF collaboration branch) | ✅ Required reviews |
| `feature/*` | New pipelines / features | Open |
| `hotfix/*` | Emergency production fixes | Open → fast-track to `main` |

### Flow

```
feature/* ──► dev ──► qa ──► main
                │       │        │
              Deploy   Deploy   Deploy
               Dev      QA      Prod
                               (manual approval)
```

## ⚙️ CI/CD Workflows

| Workflow | File | Trigger | Target |
|----------|------|---------|--------|
| Validate PR | `adf-validate.yml` | PR → `dev` or `qa` | Validation only |
| Deploy Dev | `adf-deploy-dev.yml` | Push to `dev` | Dev ADF |
| Deploy QA | `adf-deploy-qa.yml` | Push to `qa` | QA ADF |
| Deploy Prod | `adf-deploy-prod.yml` | Push to `main` + **manual approval** | Prod ADF |

Each deployment:
1. Validates the ARM template
2. **Stops** all active triggers (pre-deploy)
3. Deploys the ARM template
4. **Restarts** triggers (post-deploy)
5. *(Prod only)* Tags the release with `vYYYY.MM.DD-<run>`

## 🔒 Secrets Configuration

Go to **Settings → Secrets and variables → Actions** and add:

| Secret | Description |
|--------|-------------|
| `AZURE_CREDENTIALS` | Service principal JSON (`az ad sp create-for-rbac --sdk-auth`) |
| `AZURE_SUBSCRIPTION_ID` | Azure Subscription ID |
| `ADF_RESOURCE_GROUP_DEV` | Dev resource group |
| `ADF_RESOURCE_GROUP_QA` | QA resource group |
| `ADF_RESOURCE_GROUP_PROD` | Prod resource group |
| `ADF_FACTORY_NAME_DEV` | Dev ADF instance name |
| `ADF_FACTORY_NAME_QA` | QA ADF instance name |
| `ADF_FACTORY_NAME_PROD` | Prod ADF instance name |

> **Never** commit secrets or credentials to this repository. Use Azure Key Vault references in linked services.

## 🚀 Deploy Manually

```powershell
# Authenticate
Connect-AzAccount

# Deploy to Dev
.\scripts\deploy-adf.ps1 `
    -SubscriptionId    "<SUBSCRIPTION_ID>" `
    -ResourceGroupName "rg-adf-dev" `
    -DataFactoryName   "adf-dev-yourproject" `
    -Environment       "dev"
```

See [docs/deployment-guide.md](docs/deployment-guide.md) for full instructions.

## 🏷 Naming Conventions

| Prefix | ADF Object |
|--------|-----------|
| `PL_` | Pipeline |
| `DS_` | Dataset |
| `LS_` | Linked Service |
| `TR_` | Trigger |
| `DF_` | Data Flow |
| `IR_` | Integration Runtime |

Example: `PL_Ingest_SQLServer_To_ADLS_Daily`

## ✅ Best Practices

- **No hardcoded credentials** – all secrets via Azure Key Vault references
- **Parameterise everything** environment-specific: endpoints, paths, names
- **One PR per logical change** – keep diffs small and reviewable
- **Always update** [docs/pipeline-catalog.md](docs/pipeline-catalog.md) when adding/removing pipelines
- **Tag every production release** for easy rollback
